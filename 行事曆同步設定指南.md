# 數辦公用行事曆 — 自動同步設定指南（實際運作版）

主管在系統按下「核准 / 核定」後，系統會自動把該筆請假或加班寫入**數辦公用 Google 行事曆**。

> ✅ 本功能已於正式站設定完成並運作。本文件記錄實際採用的設定，供日後維護或重建參考。

---

## 運作原理

```
主管按「核准」
 → 前端寫入 Firestore（status 變「核准/已核准」）
 → 前端呼叫後端 /api/sync-calendar（只送 type + 文件ID + appId）
 → 後端用服務帳戶「自己去 Firestore 確認狀態確實已核准」
 → 透過網域委派，以校內帳號身分把事件寫入數辦公用行事曆
```

**安全設計**：
- 後端自我驗證 Firestore 狀態，前端無法偽造任意行事曆事件。
- 私鑰只放 Vercel 環境變數，**不進前端、不進 GitHub**。

## 一次性設定步驟（如需重建時照做）

### 步驟 0：啟用 Google Calendar API ★最容易漏★
1. 用**專案擁有者**（建立此專案的**個人 Gmail**）登入 [Google Cloud Console](https://console.cloud.google.com/)。
   > ⚠️ 若用沒有 Owner/Editor 權限的帳號（例如只開了檢視權的 ydlo 帳號），會出現「需要額外存取權 / resourcemanager.projects.get 缺少權限」，且**無法啟用 API**。
2. 開啟此連結並按「**啟用 / ENABLE**」：
   `https://console.cloud.google.com/apis/library/calendar-json.googleapis.com?project=yunlin-digital-travel`
3. 等 2～5 分鐘生效。

### 步驟 1：取得服務帳戶金鑰
- 用的是 Firebase 內建 Admin SDK 服務帳戶。
- 取得金鑰：[Firebase Console](https://console.firebase.google.com/) → 專案設定 → **服務帳戶** → 「產生新的私密金鑰」→ 下載 JSON。
- ⚠️ JSON 是機密，**不要放進專案、不要上傳 GitHub**。若不慎外洩，到 Cloud Console → IAM → 服務帳戶 → 金鑰，刪除該把並重產。

### 步驟 2：設定網域委派（繞過外部共用限制）
1. 從 JSON 取得 **`client_id`**（純數字，例：`104264150357341794024`）。
2. 用 **ydlo 超級管理員**登入 `admin.google.com` →
   **安全性 → 存取權與資料控管 → API 控管 → 網域委派 → 新增**：
   - 用戶端 ID：上面的 `client_id`
   - OAuth 範圍：`https://www.googleapis.com/auth/calendar.events`
3. 授權後等幾分鐘生效。

### 步驟 3：確認「假扮帳號」對行事曆有編輯權
- 假扮帳號（`GOOGLE_IMPERSONATE_SUBJECT`，目前為 `jackize705@ydlo.ylc.edu.tw`）
  必須對「數辦公用行事曆」有「變更活動」以上權限（用該帳號登入行事曆能看到並能編輯即可）。

### 步驟 4：取得行事曆 ID
- 該行事曆 → 設定 → 「整合行事曆」→ 複製「行事曆 ID」。
- 目前：`c_7fe0f75e56b00fbc2af3d5860b17672539c790cfd7417b464782c31f71f4e63a@group.calendar.google.com`

### 步驟 5：在 Vercel 設定環境變數（5 個，環境都勾 Production）
| 變數 | 值 |
|---|---|
| `GOOGLE_PROJECT_ID` | `yunlin-digital-travel` |
| `GOOGLE_CLIENT_EMAIL` | `firebase-adminsdk-fbsvc@yunlin-digital-travel.iam.gserviceaccount.com` |
| `GOOGLE_PRIVATE_KEY` | JSON 的 `private_key` 整段（含 BEGIN/END，`\n` 原樣保留，不含外層引號） |
| `OFFICE_CALENDAR_ID` | 上面的行事曆 ID |
| `GOOGLE_IMPERSONATE_SUBJECT` | `jackize705@ydlo.ylc.edu.tw` |

> 改完環境變數要**重新部署**才生效（Deployments → 最新一筆 → Redeploy）。

## 行事曆呈現方式

- 事件以**全天**顯示（避免時數非整天時在行事曆上變成一個小點）。
- 實際時間放在「**備註**」第一行，例如：`🕒 時間：2026-06-30 13:00 ~ 2026-06-30 17:00`。
- 多日請假：起日到迄日整段以全天區塊呈現。

## 測試與驗證

1. 用同仁帳號送一筆**測試**請假。
2. 用主管帳號**核准**。
3. 看到提示「**已自動同步至數辦公用行事曆！**」。
4. 打開公用行事曆，確認出現全天事件、備註內時間正確。

## 常見錯誤排查（皆為實際遇過）

| 訊息 | 原因與處理 |
|---|---|
| `Calendar API has not been used ... or it is disabled` | 沒啟用 Calendar API（步驟 0）；注意要用**擁有者帳號**啟用 |
| 「需要額外存取權 / projectMover」 | 登入的帳號沒有此專案權限；換成**擁有者個人 Gmail**操作，並忽略它建議的 projectMover 角色 |
| `unauthorized_client` | 網域委派未生效或 client_id/scope 沒對上（步驟 2），等幾分鐘或重設 |
| `403 / forbidden` 寫行事曆 | 假扮帳號對行事曆沒編輯權（步驟 3） |
| `Requested entity was not found` | `OFFICE_CALENDAR_ID` 填錯 |
| `找不到該筆記錄` | 該文件 ID 不存在（正常防呆） |
| `... is reserved` | 文件 ID 用了前後雙底線等保留字（Firestore 限制） |

## 回復（萬一要還原）

- **線上立即回復**：Vercel → Deployments → 選上一個正常版本 → Promote / Rollback（幾秒生效，不碰資料）。
- **程式碼回復**：GitHub 可 revert；本機 git 亦有提交快照。
- **資料安全**：差勤紀錄存在 Firebase Firestore，與程式碼部署完全分離，部署不會影響既有資料。
